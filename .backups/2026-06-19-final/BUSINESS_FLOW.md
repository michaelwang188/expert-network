## 三、状态流转图

```
PENDING ──→ ACTIVE ──→ DONE ──→ PAID
   │          │                   ↑
   │          │          (管理员确认结算)
   │          │
   └──── CANCELLED ←───┘
     (任意时刻取消)
```

| 状态 | 含义 | 谁在操作 |
|------|------|---------|
| **PENDING** | 待派单 或 待专家确认 | 管理员派单 / 专家接单 |
| **ACTIVE** | 访谈进行中 | 专家/管理员标记完成 |
| **DONE** | 访谈已完成，待结算 | 管理员确认结算 |
| **PAID** | 已结算（终结状态） | — |
| **CANCELLED** | 已取消（终结状态） | 研究员/管理员取消 |

## 五、管理员在 /admin 和 /orders 的角色

**管理员在订单流程中承担两个角色：**

1. **派单调度员**（/admin）：研究员提交需求后没有专家 → 管理员人工匹配专家并派单。这是运营的核心环节。

2. **结算审核员**（/orders）：访谈完成后 → 管理员确认结算。结算时系统记录积分变动。这是风控的最后一道门。

---

## 六、积分变动规则（结算时触发）

| 角色 | 变动方向 | 说明 |
|------|---------|------|
| 研究员 | **扣减** | `Order.amount` 积分（`SPEND_ORDER`） |
| 专家 | **增加** | `Order.expertFee` 积分（`EARN_LABOR`） |
| 平台 | — | `Order.platformFee` 积分（平台服务费，不进任何个人账户） |
