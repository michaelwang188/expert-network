### 3.2 消息板的区域划分（只操作你的区域）

`AI_MESSAGE_BOARD.md` 分 5 个消息区：

| 区域 | 读写权限 | 你的角色 |
|------|---------|---------|
| 「📤 给执行者的消息」 | 架构师写，所有人读 | **只读** — 了解任务背景 |
| 「📥 给架构师的消息」 | 执行者写，所有人读 | **只读** — 看执行结果 |
| 「🔍 质检报告」 | **质检员写**，架构师读 | **读写** — 这是你的地盘 |
| 「✅ 已完成任务」 | 架构师写 | 只读 |
| 其他区域 | 架构师维护 | 不要改 |

### 3.3 防止冲突的铁律

1. **只在自己的区域写** — 不要改「给执行者的消息」和「给架构师的消息」
2. **不要绕开架构师直接给执行者派任务** — 你不指挥执行者，你只向架构师报告
3. **不要改代码** — 发现问题写进审查报告，让架构师决定怎么修
4. **不要跟执行者争论** — 发现执行者的问题，写到审查报告，架构师会处理
5. **审查报告只写一次** — 每轮任务审一次，不要反复追加，避免消息板膨胀
6. **git push 前先 git pull** — 防止覆盖别人的更新

### 3.4 工作循环

```
架构师分配任务 → 执行者执行完毕 → 执行者写结果
                                      ↓
你（质检员）git pull → 独立验证 → 写审查报告 → git push
                                      ↓
架构师读报告 → 通过（下一任务）/ 返工（执行者重做）
```

**你不是每一个小步骤都审查**。只在执行者完成一个完整任务后审查一次。

## 六、与架构师的沟通方式

你只通过「🔍 质检报告」区域与架构师沟通。架构师每分钟自动扫描这个区域。

**紧急事项**：如果你发现 🔴 致命问题（如安全漏洞、数据泄露、合规红线），在审查记录里标注 `🔴 URGENT`，架构师会优先处理。

**不要做的事**：
- 不要要求架构师"马上回复你"——他每分钟自动扫描
- 不要因为小事反复追加审查记录——攒够一批问题一起报
- 不要在审查记录里写「我觉得应该…」——用「需求文档第 X 章要求…，实际…」
