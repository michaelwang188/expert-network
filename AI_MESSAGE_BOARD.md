## 第一篇：发生了什么变化（重要！）

### 旧模式（已废止 ❌）

```
4个AI通过消息板15秒轮询协作
autopilot_v4.py 后台守护
Mavis 通过 DeepSeek→Anthropic 代理API 做质检（长期不通）
1号(文案) → 2号(执行) → 3号(质检) → 4号(审计) → 消息板 → 循环
```

**问题：** API不通导致Mavis瘫痪、一个功能4个人接力耗时是直接干的4倍、163个task文档但关键功能没进展、992次commit大部分是空转。

### 新模式（当前 ✅）

```
WorkBuddy(产品/方向) → 写spec → git push → 消息板
        ↓
Claude(执行) → 读到任务 → 直接干 → 写完成报告 → git push
        ↓
Mavis(测试) → 读到完成报告 → 浏览器仿真测试 → 写测试报告 → git push
        ↓
用户(决策) → 看结果 → 说行/不行 → 推进下一步
```

**核心变化：**
1. **不要autopilot_v4.py** — 已停止。不要重启。
2. **不要15秒轮询** — 改为 `stat` 文件监听（每15秒检查mtime，零算力，弹macOS通知）
3. **不要API接力** — Mavis用Trae内置浏览器做测试，不需要任何API Key
4. **一个人干完一个功能** — 不再分1/2/3/4号，谁负责就干到底
5. **任务成果交付到消息板** — 干完写完成报告，不是把代码扔那就完事

## 第三篇：消息板机制说明

### 目录结构

| 区域 | 谁写 | 谁读 | 用途 |
|:-----|:----:|:----:|:------|
| `🔥 当前任务` | WorkBuddy | Claude | 优先级排序的任务清单 |
| `📥 WorkBuddy→Claude` | WorkBuddy | Claude | 分配给Claude的任务、策略指示 |
| `📤 Claude→WorkBuddy` | Claude | WorkBuddy | 完成报告、需要决策的问题 |
| `🧪 Mavis测试报告` | Mavis | Claude+WorkBuddy | 测试结果、Bug报告 |

### 消息板规则

1. **写消息必须带时间戳标签**：`### 06-23 19:00 | 发送者 → 接收者`
2. **每个新任务/回复前加分隔线 `---`**
3. **完成的旧消息不要删除**，让后到的人能追溯上下文
4. **不要写"收到""好的"这类空消息**，消息板不是聊天工具
5. **任务完成后必须写完成报告**，格式如下：

```
### 06-23 20:00 | Claude → WorkBuddy

**主题：P0 结算自动创建PointsTransaction 完成报告**

完成内容：
- 修改了 xx 文件，增加了 xx 逻辑
- 测试方法：curl 测试了 xx 场景
- 结果：通过

待确认问题：
- xxx 需要用户决定

下步计划：
- 开始 P1
```

### 消息板监听器

```
脚本位置：~/.claude/wechat/scripts/message_board_watcher.sh
运行状态：✅ 已启动（PID: 62508）
检查频率：每15秒
触发方式：文件mtime变化 → macOS通知弹窗
算力消耗：零（纯本地 stat 命令）
```

不需要自己跑轮询脚本。不需要自己检查消息板。**写完 push 后15秒内，所有人都会收到通知。**

## 第六篇：常见问题

**Q: 消息板要不要实时在线？**
A: 不需要。写完push就行，监听器会通知其他人。

**Q: 我能不能改别人的代码？**
A: 各司其职。WorkBuddy不改代码，Mavis不改代码，Claude不改需求和设计。有异议写到消息板讨论。

**Q: 新旧消息怎么区分？**
A: 看时间戳。06-23 18:00之前的消息是旧模式的，可以忽略。之后的才是当前有效的。

**Q: 监听器如果停了怎么办？**
A: 监听器进程文件在 `/tmp/msg_watch.sh`，重启命令：
```bash
kill $(pgrep -f msg_watch) 2>/dev/null
nohup /tmp/msg_watch.sh > /dev/null 2>&1 &
```

**Q: 我发现了Bug要怎么办？**
A: 写到消息板 `🧪` 区（如果你是Mavis）或 `📥` 区（如果你是WorkBuddy），标记清楚优先级和复现步骤。
